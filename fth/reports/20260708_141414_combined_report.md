## Key Findings

### 🔴 Upload — large multipart objects fail with `BadDigest` (3/5)

All three failures are ~21 MB `.zip` objects pulled from the `harvard-lil` source bucket; the server rejects them with HTTP 400 `BadDigest` **after 4 retries** (185–260 s each). The two successes were small objects (5.5 KB and 740 KB).

| Key (suffix)                      |    Size | Elapsed | Result       |
| --------------------------------- | ------: | ------: | :----------- |
| `README.md`                       |  5.5 KB |    2.0s | ✅ ok        |
| `-511-event-extents-6f436/v1.zip` |  740 KB |    3.5s | ✅ ok        |
| `…ex-0397e/v1.zip`                | 21.0 MB |  260.1s | ❌ BadDigest |
| `…ex-54c47/v1.zip`                | 21.1 MB |  239.1s | ❌ BadDigest |
| `…ex-aa506/v1.zip`                | 21.1 MB |  185.5s | ❌ BadDigest |

**Signal:** the size threshold (small pass, ~21 MB fail) points at multipart-upload checksum handling — the computed content MD5 / checksum doesn't match what the server derives. Worth checking whether the client is streaming a chunked/multipart body while sending a whole-object `Content-MD5`, and whether the `via: 1.1 Caddy` proxy is altering the body.

### 🔴 Presign — browser CORS response missing expose-headers (3/24)

The presign pipeline (sign → preflight → execute) succeeds for every operation; the failures are all `ResponseCorsMissing` on the **response CORS** check — the `Access-Control-Expose-Headers` list omits headers the browser needs:

| Operation                         | Missing expose-headers                       |
| --------------------------------- | -------------------------------------------- |
| `response_cors_getObject`         | `content-type`, `last-modified`              |
| `response_cors_headObject`        | `content-type`, `last-modified`              |
| `response_cors_headObjectFilMeta` | `content-type`, `last-modified`, `x-fil-cid` |

`putObject`, `listObjects`, `deleteObject`, and `getObjectRetention` response-CORS checks all pass. Fix is server-side: add `Content-Type`, `Last-Modified` (and `x-fil-cid` for the Fil-meta path) to the bucket's `Access-Control-Expose-Headers`.

### 🟡 Presign — skipped: cannot read bucket CORS (1/24)

`bucket_cors_read_pre` was skipped — `GetBucketCors` returns `AccessDenied`, so the test can't verify the bucket's CORS config directly (provider manages CORS out-of-band). Informational, not a failure.

### 🟢 Fetch — all pass

`head_object`, `get_object_preview`, and `list_versions` all succeeded across 2 keys, sub-600 ms.

### 🟢 Delete — all pass

Both `delete_object` calls succeeded (0.33s, 0.63s).

## Recommended Actions

1. **Upload `BadDigest`** — reproduce with a single ~21 MB object and inspect the multipart checksum path; confirm whether the Caddy proxy in front of the endpoint mutates the body. Highest priority (data-write failures).
2. **Presign response CORS** — extend `Access-Control-Expose-Headers` to include `Content-Type`, `Last-Modified`, and `x-fil-cid`. Purely a header-config change; unblocks browser GET/HEAD.
3. **Bucket CORS read** — grant the test principal `s3:GetBucketCors` (or accept the skip) so CORS config can be asserted directly rather than inferred.
