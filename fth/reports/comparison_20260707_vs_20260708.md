compat-test key** — the top recommendation from the prior analysis. This doc
measures what that change bought and what still fails.

## Analysis

### What the new keys fixed (the +43)

`AccessDenied` fell 61 → 20. The recovered tests are exactly the categories the
prior doc attributed to under-scoped keys:

- **tagging 2 → 20** (+18) — Get/Put/DeleteObjectTagging now permitted.
- **s3_core +17** — object/bucket ACL gets, `ListBucketMultipartUploads` /
  `AbortMultipartUpload`, public-access-block and lifecycle _gets_.
- **bucket_policy 2 → 6**, **lifecycle 12 → 14**, **lifecycle_expiration 4 → 6**.

This confirms root cause #1 from the prior analysis: the key set, not the
provider, was behind these.

### Why it wasn't the full 80–90: keys still incomplete

20 `AccessDenied` remain, and they cluster on actions the _new_ set still lacks
— mostly `Delete*`-config and ACL/multipart write:

| Operation                                                            |  Count |
| -------------------------------------------------------------------- | -----: |
| `PutObject` (canned-ACL puts)                                        |      5 |
| `DeletePublicAccessBlock`                                            |      3 |
| `DeleteBucketEncryption`                                             |      2 |
| `DeleteBucketTagging` / `DeleteBucketLifecycle` / `DeleteBucketCors` | 1 each |
| `CreateMultipartUpload`                                              |      1 |
| (remainder surface as downstream assertions)                         |     ~6 |

So the re-provision added the read/write config actions but missed the matching
**`Delete*` config actions** and **`PutObject` ACL / some multipart** — a second,
smaller permission pass would recover most of these.

### What keys can't fix — provider bugs, all persisted verbatim

None of these moved between Jul 7 and Jul 8, confirming they are server-side and
independent of credentials:

1. **🐞 Zero-byte PutObject → HTTP 500 `InternalError` (47 tests).** Unchanged
   and still the single largest cluster. Every test uploading an empty/omitted
   `Body` (e.g. `test_basic_key_count`, `test_object_head_zero_bytes`,
   `test_atomic_dual_write_*`, `test_object_raw_*`). Highest-impact fix for FTH.
2. **🐞 SSE-KMS rejected with `InvalidRequest` (39).** All KMS Put/Copy/CreateMPU.
   Keeps `encryption` at 63/108 and drags `copy`. Likely no KMS key provisioned
   for the tenant, or dropped SSE-KMS support — verify server-side.
3. **🐞 `SignatureDoesNotMatch` on special-character keys + SigV2 presign (21).**
   Keys with `+`, space, `%`; plus SigV2 `CreateBucket`. boto signs correctly →
   server canonicalization regression.
4. **🐞 `PutBucketLogging` → `InvalidArgument` (6).** `bucket_logging` still 0/6.
5. **🐞 Conditional `PutObject` → `PreconditionFailed` (3).** `If-Match`/`If-None-Match`.
6. **🐞 `BadDigest` on large PUT/copy (2).** Matches the standalone upload
   regression on ~16–21 MB objects — same multipart-checksum smell.

### ⚠️ New minor regression: object_lock 39 → 37

Object lock was fully green on both July runs; the new run shows **2 errors**
out of 39 (the run's only 2 `Error`-status results, both in the quarantine
pass). These are _errors_ (setup/teardown) rather than assertion failures —
worth a glance to confirm they're not fallout from the key change, but low
severity.

## Recommendation

1. **Second key pass** — add the missing `Delete*`-config actions
   (`DeletePublicAccessBlock`, `DeleteBucketEncryption`, `DeleteBucketCors`,
   `DeleteBucketTagging`, `DeleteBucketLifecycle`), plus `PutObjectAcl`/canned-ACL
   `PutObject` and the remaining multipart action. Recovers most of the last 20
   `AccessDenied`. Low effort.
2. **File the provider bugs with FTH** (unchanged, ~118 tests, credential-independent):
   zero-byte PutObject 500 (47), SSE-KMS `InvalidRequest` (39), special-char /
   SigV2 signature (21), PutBucketLogging (6), conditional-write (3), BadDigest (2).
   The zero-byte 500 is the highest-leverage single fix.
3. **Triage the 2 object_lock errors** to rule out a key-related setup regression.
