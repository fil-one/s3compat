## Root causes

The credentials changed between the runs: `.env` vs `.env.old` show the same
endpoint but **new main and alt access keys** for the July run. That change is
the single biggest contributor.

### 1. Test configuration — under-scoped access keys (~60 AccessDenied failures + most assertion failures)

The new keys were minted via `bin/create-fth-tenant.ts`, which grants
`FTH_FULL_PERMISSIONS` (defined in
`packages/backend/src/lib/fth/fth-tenant-setup.ts`). Despite the name, that set
contains only 19 actions — bucket/object CRUD, versioning, and object lock. It
is missing everything else the suite exercises:

- `s3:GetBucketAcl` / `s3:PutBucketAcl`, `s3:GetObjectAcl` / `s3:PutObjectAcl`
- `s3:GetObjectTagging` / `s3:PutObjectTagging` / `s3:DeleteObjectTagging`
- `s3:GetBucketPolicy` / `s3:PutBucketPolicy` / `s3:DeleteBucketPolicy`, `s3:GetBucketPolicyStatus`
- `s3:GetBucketPublicAccessBlock` / `s3:PutBucketPublicAccessBlock` / delete
- `s3:GetLifecycleConfiguration` / `s3:PutLifecycleConfiguration`
- `s3:GetEncryptionConfiguration` / `s3:PutEncryptionConfiguration` / delete
- `s3:AbortMultipartUpload`, `s3:ListBucketMultipartUploads`

**Smoking gun:** `test_bucket_acl_default` creates a bucket with the _main_
client and immediately gets `AccessDenied` calling `GetBucketAcl` on its own
bucket. This explains tagging collapsing 22→2, bucket_policy 11→2, and the
ACL / public-access-block / lifecycle-get / multipart abort-and-list clusters.

### 2. Provider bug — zero-byte PutObject returns InternalError (47 tests)

Every test in this cluster uploads an object with an empty or omitted `Body`
(`test_object_head_zero_bytes`, `test_basic_key_count`,
`test_atomic_dual_write_*`, all `test_object_raw_*` setups, the presigned CORS
setups, `test_ranged_request_empty_object`, …) and receives HTTP 500
(`InternalError`, retried 4×). All of these passed in April. **Not a config
issue** — an FTH regression on 0-byte PUTs, and the most impactful single item.

### 3. Provider bug — SignatureDoesNotMatch on special-character keys (19 tests)

Keys containing `+`, space, or `%`
(`test_bucket_list_delimiter_percentage` / `_whitespace`,
`test_bucket_create_special_key_names`, encoding tests, …) now fail signature
validation. The SigV2 presigned CORS tests additionally fail on a plain
`CreateBucket`. boto signs these correctly, so this is a server-side
canonicalization / SigV2 regression. All passed in April.

### 4. SSE-KMS rejected with InvalidRequest (38 tests)

Every `sse-kms` PutObject / CopyObject / CreateMultipartUpload is refused.
These passed in April with an identical harness config (the generated
`s3tests.conf` never set a `kms_keyid` in either run), so either FTH
dropped/changed SSE-KMS handling, or the **new tenant** has no KMS key
provisioned server-side. Worth checking during re-provisioning — this one
straddles the config/provider line.

### 5. Small clusters

- **bucket_logging 6→0:** `InvalidArgument: Invalid bucket logging destination` — provider behavior change.
- **conditional_write 7→3:** `If-Match` / `If-None-Match` PutObject now returns `PreconditionFailed` where it used to succeed.
- **One `BadDigest`** on a 16 MB PutObject (`test_object_copy_16m`).

multipart abort/list are all authorized now.

### What's left, in order of size

1. **Provider bugs, unchanged from Jul 6/7** (~110 tests): zero-byte
   PutObject `InternalError` (47), SSE-KMS `InvalidRequest` (39),
   `SignatureDoesNotMatch` on special-character keys and SigV2 presign (~21),
   bucket logging `InvalidArgument` (6), conditional-write
   `PreconditionFailed` (3), `BadDigest` on 16 MB PUT.
2. **Missing `Delete*` permissions (~8 tests):** FTH models delete
   operations as distinct permissions rather than folding them into the
   `Put*` action as AWS does. Still denied: `DeletePublicAccessBlock`,
   `DeleteBucketEncryption`, `DeleteBucketLifecycle`, `DeleteBucketTagging`,
   `DeleteBucketCors`. Fix: add the corresponding `s3:Delete*` actions to
   `FTH_COMPAT_TEST_PERMISSIONS` in `bin/create-fth-tenant.ts`.
3. **Cross-user tests are not meaningful with the current key setup
   (19 tests):** `AssertionError: ClientError not raised` — the alt client
   can read the main client's _private_ objects
   (`test_access_bucket_private_object_private` et al.). This is expected,
   not an FTH defect: both compat keys belong to the **same storage user**,
   so the "alt" client is the same principal and shared visibility is
   correct. To make these 19 tests meaningful, provision the alt key on a
   separate storage user (what `bin/create-fth-tenant.ts` now does) and
   rerun; only then would a still-passing read of a private object indicate
   an isolation problem.
4. **Quarantine pass regressed slightly: 37/39** (was 39/39 on Jul 6/7).
   The two object-lock mode-change tests again fail teardown with
   `BucketNotEmpty` on DeleteBucket — the same teardown errors April had.

## Object lock (quarantine pass)

The object-lock tests run in a separate quarantine pytest pass, and their
results are folded into `s3_core` by the report generator rather than shown
as their own category. Extracted from
`*_compatibility_pytest_quarantine.json`:

|             | Apr 21 | Jul 6 | Jul 7 |
| ----------- | -----: | ----: | ----: |
| object_lock |  33/39 | 39/39 | 39/39 |

This category **fully recovered** — the six April failures all pass in both
July runs (identical results on Jul 6 and Jul 7):

| Test                                                         | April failure                                                    |
| ------------------------------------------------------------ | ---------------------------------------------------------------- |
| `test_object_lock_put_obj_lock_invalid_bucket`               | HTTP 502 instead of 409                                          |
| `test_object_lock_put_obj_lock_enable_after_create`          | HTTP 502 instead of 409                                          |
| `test_object_lock_suspend_versioning`                        | HTTP 502 instead of 409                                          |
| `test_object_lock_get_obj_lock_invalid_bucket`               | `NoSuchBucket` instead of `ObjectLockConfigurationNotFoundError` |
| `test_object_lock_changing_mode_from_governance_with_bypass` | teardown: `BucketNotEmpty` on DeleteBucket                       |
| `test_object_lock_changing_mode_from_compliance`             | teardown: `BucketNotEmpty` on DeleteBucket                       |

Notably, object lock is one of the few areas the narrow key-permission set
actually covers (`s3:*ObjectLockConfiguration`, `*ObjectRetention`,
`*ObjectLegalHold` are all in `FTH_FULL_PERMISSIONS`), which is consistent
with it being unaffected by the AccessDenied regression that hit the other
categories.

## Improvements since April (25 tests fixed)

- Metadata key casing (`Foo` vs `foo`) — all multipart/copy/SSE metadata tests
- Object-lock error codes and status codes (was 502, now correct)
- SSE-C multipart metadata tests
- `test_100_continue`, `test_list_buckets_paginated`, `test_get_multipart_checksum_object_attributes`

## Recommendation

1. **Re-provision the compat-test key with a broader action set** — either add
   the missing actions above to a dedicated `FTH_COMPAT_TEST_PERMISSIONS` set
   (preferable to widening the console key) and rerun. Expected recovery:
   roughly 80–90 tests.
2. **Report the remaining regressions to FTH:** zero-byte PutObject 500s (47
   tests), special-character key signature failures, SSE-KMS rejection, bucket
   logging destination validation, and conditional-write regressions.
